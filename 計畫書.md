# LDCT 肺癌篩檢個案管理系統 — 計畫書

本文件與 OpenSpec 變更 `add-ldct-case-management-platform` 對齊；實作程式或調整範圍時應同步修訂本計畫書與 [openspec/changes/add-ldct-case-management-platform/](openspec/changes/add-ldct-case-management-platform/) 內規格。

**系統分析與設計（技術文件）**：[docs/SA.md](docs/SA.md)（SA：業務需求、外部介面、與 OpenSpec 對照）、[docs/SD.md](docs/SD.md)（SD：邏輯架構、整合與資料流）；**專案脈絡與外部系統表**見 [openspec/project.md](openspec/project.md)。需求驗收仍以 OpenSpec `specs` 為準；計畫書、SA、SD、`project.md` 敘述應一致，變更時一併更新。

## 一、計畫目標

建立 LDCT 篩檢個案追蹤管理平台，整合健檢資料、影像報告、問卷與追蹤紀錄，協助個案管理師管理個案、自動產生追蹤名單、發送簡訊提醒、記錄電訪與追蹤結果、**於系統內連結 HIS 確認相關就醫資訊**、監測確診案例；**統計與報表由本平台提供報表資料集，以 Microsoft Power BI 完成視覺化與月報／管理儀表**（與 [SA.md](docs/SA.md) 第 4 節業務目標、[SD.md](docs/SD.md) 設計原則一致）。

## 二、資料來源

（業務向摘要；明細與責任劃分亦見 [SA.md](docs/SA.md) 第 6 節。）

| 來源           | 內容摘要                         |
| ------------ | ---------------------------- |
| 健檢系統名單       | 病歷號、姓名、性別、年齡、檢查日期、客戶名稱、電話、手機 |
| LDCT 報告（EHR，主來源） | 是否有結節、結節數量、最大長度、門診追蹤、3／6／12 個月追蹤；非結構化內容時以 **LLM 擷取** 結構化欄位並供人工覆核 |
| 個管師輸入        | 判讀補充、追蹤狀態、電訪內容、聯繫紀錄          |
| 問卷系統         | 肺癌風險評估問卷                     |
| 個管師（佐證 HIS）  | 經本系統**連結 HIS** 查詢後登錄或註記；確診與病理、手術、Stage 等 |

**LDCT 報告落地時機（院方共識）**：報告於 **EHR 已正式發佈（已發報告）** 之後，於**同一曆日之晚間時段**（排程批次，具體鐘點與時區、假日規則與失敗重試見 [design.md](openspec/changes/add-ldct-case-management-platform/design.md)「資料落地與同步時機」）將報告資料**落地**至本平台，並使該筆個案資料可供**個案總清單**使用（須已能與健檢名單等來源正確關聯；驗收見 OpenSpec `data-sources`）。

## 三、核心功能（對應規格 capability）

### 需求功能清單簡表

| 編號 | 分類 | 需求功能 | OpenSpec capability | 摘要 |
|------|------|----------|---------------------|------|
| 3.1 | function | 個案清單 | `case-list` | 總清單欄位、查詢／篩選（含本月應追蹤預設）、列尾四連結、檢視報告、結案觸發 |
| 3.2 | function | 追蹤紀錄 | `follow-up-logging` | 個案脈絡載入、事項登錄（軌道／聯繫類型／日期／結果／內容）、列表排序、補正稽核、與 `sms-fetnet`／HIS 註記對應；總清單「追蹤記錄」鈕固定導向本項 |
| 3.3 | function | 簡訊 | `sms-fetnet` | 遠傳公務機發送與紀錄；列尾「簡訊發送」可導向本流程 |
| 3.4 | function | 結案 | `case-closure` | 結案代碼與流程；與清單及統計口徑一致 |
| 3.5 | function | 臨床提醒 | `clinical-alerts` | 病理關鍵字、切片、逾期等提醒 |
| 3.6 | report | 統計報表 | `reporting`、`system-integrations`（Power BI） | 本平台提供報表資料集；**Microsoft Power BI** 視覺化與月報／儀表 |
| 3.7 | function | 角色 | `rbac` | 角色權限與列尾連結等操作控管 |
| 3.8 | integration | 整合 | `system-integrations`、`data-sources` | 外部系統介接與資料來源（健檢、EHR、LLM、HIS、問卷等）；細部見第二節與 OpenSpec `data-sources` |

**分類**：`function`＝業務／作業功能；`report`＝報表資料集與 **Power BI** 等分析呈現；`integration`＝外部系統與資料來源整合。

（資料匯入與欄位之**單一規格**亦見 `data-sources`；驗收以 `openspec/changes/add-ldct-case-management-platform/specs/**/spec.md` 為準。）  
（本節計畫書**不**另列：`follow-up-lists`、`follow-up-workflow` 之獨立明細，以及個案清單之「加入追蹤」子項；OpenSpec 已於該二 capability 標示 **REMOVED Requirements**，並同步 [docs/SA.md](docs/SA.md)、[docs/SD.md](docs/SD.md)、[proposal.md](openspec/changes/add-ldct-case-management-platform/proposal.md)。）

本節功能以**三層十進位編號**對應章節「三」：`3.*`（第一層：capability）、`3.*.*`（第二層：子項）、`3.*.*.*`（第三層：細項）。以下為明細。

3.1 **個案清單**（`case-list`）  
　3.1.1 **總清單欄位**：病歷號、姓名、檢查日期、客戶名稱、追蹤類型、**追蹤1、追蹤2、追蹤3、追蹤4**、胸腔門診、3 months、6 months、12 months、LDCT 狀態、結案狀態（追蹤1～4 之語意如追蹤次序日期或狀態，依院方欄位定義；與 `case-list` 規格一致）。  
　3.1.2 **查詢與篩選**  
　　　3.1.2.1 **條件與日期範圍**：依條件查詢個案；依**日期範圍**篩選（檢查日期或追蹤日期等，依院方定義之主要篩選欄位）。  
　　　3.1.2.2 **預設篩選（本月應追蹤）**：以**檢查日期**為基準，依報告欄位 **胸腔門診（1 個月）**、**3 months**、**6 months**、**12 months** 是否為 true（或等價旗標），計算各類追蹤應到日；若**至少一類**之應到日落在**當月**，則預設顯示該筆（與未結案等條件一致；可改為自訂篩選）。詳見 [design.md](openspec/changes/add-ldct-case-management-platform/design.md)「個案／追蹤清單預設篩選」。  
　3.1.3 **列尾捷徑**：總清單**每列最後**提供四個連結按鈕——**追蹤記錄**（**固定**導入 **`follow-up-logging`** 追蹤紀錄功能）、**報告查詢**、**基本資料**、**簡訊發送**（後三者之出站 URL 或站內畫面依院方設定與 `rbac`）；說明見同一 [design.md](openspec/changes/add-ldct-case-management-platform/design.md)「個案總清單列尾連結」。  
　3.1.4 **檢視報告**：自清單檢視該個案之 LDCT 報告內容或連結；若全文存於 EHR 或外部影像系統，至少提供摘要與外部連結或院方約定之開啟方式。  
　3.1.5 **結案**：自清單觸發結案流程，更新結案狀態；已結案者於未結案追蹤統計中排除。  
3.2 **追蹤紀錄**（`follow-up-logging`）  
　3.2.1 **個案脈絡**：自清單列尾**追蹤記錄**或已授權入口進入時，依個案識別載入；畫面至少顯示病歷號、姓名、檢查日期，且僅顯示該個案之紀錄（與 `rbac` 一致）。  
　3.2.2 **追蹤事項登錄**：逐筆紀錄至少含—**追蹤軌道／類型**（對齊胸腔門診／3／6／12 個月等）、**聯繫／動作類型**（簡訊、電訪、其他）、**聯繫／追蹤日期**、選填**聯繫結果**、**追蹤內容**；並自動附**建立者、建立時間**。  
　3.2.3 **列表與排序**：以表格或時間軸呈現歷次紀錄；支援依聯繫日期或建立時間排序（預設依院方慣例）；顯示序號（第 N 次）、類型、日期與內容摘要；無紀錄時顯示空狀態。  
　3.2.4 **補正與稽核**：允許補正敘述／結果／日期等院方開放欄位；每次補正記錄修改者、修改時間，且修改前內容可追溯（策略見 OpenSpec）。  
　3.2.5 **與簡訊對應**：經 **`sms-fetnet`** 發送成功者，應與追蹤紀錄可對應（自動寫入「簡訊」類型紀錄並帶發送紀錄識別，或發送後一鍵寫入；院方／實作定案）。  
　3.2.6 **HIS 註記**：可於紀錄中註記 HIS 就醫查詢摘要、查詢日期或參考編號，與**連結 HIS**（`system-integrations`）作業銜接。  
詳見 [spec.md](openspec/changes/add-ldct-case-management-platform/specs/follow-up-logging/spec.md) 與 [design.md](openspec/changes/add-ldct-case-management-platform/design.md)「追蹤紀錄」。  
3.3 **簡訊**（`sms-fetnet`）  
3.4 **結案**（`case-closure`）  
3.5 **臨床提醒**（`clinical-alerts`）  
3.6 **統計報表**（報表資料集 + Power BI：`reporting`、`system-integrations`）  
3.7 **角色**（`rbac`）  
3.8 **整合**（`system-integrations`、`data-sources`）

## 四、技術架構（摘要）

- **專案／雛型**：**單一解決方案或儲存庫**內 **前後端分離**—後端 **`src/Api`**（**ASP.NET Core Web API**），前端 **`src/Web`**（**Vue 3**、**TypeScript**；以 API 整合，子專與建置流程見 `design.md`）。
- **資料庫**：**Microsoft SQL Server**。
- **部署目標**：**Kubernetes（K8s）**（服務拆分、Ingress、組態與 Secret 等與院方 IT 定案）。
- **其餘**：**LLM** 用於非結構化 LDCT 報告之欄位擷取（服務型態須經院方資安核定）；**Power BI** 連線本平台**報表資料集**產製月報與儀表板；叢集拓撲與備援 **RPO／RTO** 實作階段與院方 IT 定案。

敘述與 [openspec/project.md](openspec/project.md)「技術架構」「外部系統」及 [docs/SD.md](docs/SD.md) 第 3、4 節一致。外部系統之**業務介面**摘要見 [docs/SA.md](docs/SA.md) 第 7 節；**邏輯模組與資料流**見 [docs/SD.md](docs/SD.md) 第 4～6 節；變更內**整合型態**見 [design.md](openspec/changes/add-ldct-case-management-platform/design.md)。

## 五、時程與交付物

| 里程碑 | 目標週期（相對） | 交付物／驗收焦點 |
|--------|------------------|------------------|
| M0 基盤 | 第 1 階段 | 程式庫 `src/Api`、`src/Web`；JWT 雛型、EF 遷移、CI、docker-compose（SQL+API）、`/health` |
| M1 名單／清單 | 第 2 階段 | 名單匯入、晚間批次 API、個案清單 UI（本月應追蹤、列尾四連結、檢視報告、結案） |
| M2 追蹤／簡訊／結案 | 第 3 階段 | 追蹤紀錄 CRUD、簡訊模擬與紀錄連動、結案代碼 0–6 |
| M3 整合 | 第 4 階段 | HIS URL、LLM 擷取雛型、問卷 sync stub；真 PHI／正式端點須資安核定 |
| M4 提醒／報表 | 第 5 階段 | 臨床提醒規則雛型、報表 JSON 資料集（Power BI 銜接文件／連線由資訊室定案） |
| M5 上線 | 第 6 階段 | `k8s/` 範例清單（Deployment、Ingress、PDB、HPA、Secret 範例）；RPO／RTO 與院方 IT 定案；**教育訓練**（個管、護理、醫師、管理者） |

**測試環境**：建議與正式叢集同類之命名空間，連線測試用 SQL／mock 外部系統；**上線日**由院方專案辦公室公告後填入專案管制表（本表可再連結該表）。**測試主機（雛型）**：`10.41.100.1` — 瀏覽器存取前端開發伺服器為 `http://10.41.100.1:5173`（Vite `host: true`）；API 建議以 **`dotnet run --launch-profile http-lan`** 綁定 `0.0.0.0:5003`，或於該主機以 **docker-compose** 對外對應 **5003**；CORS 已納入 `http://10.41.100.1:5173`、`http://10.41.100.1:4173`（與 `appsettings.json`／`Program.cs` 預設一致）。若 API 非本機，可於 `src/Web` 建立 `.env` 設定 `VITE_API_PROXY_TARGET` 指向實際 API 基底位址。**雛型示範資料**：個案總清單之「載入示範名單與晚間批次」可一次寫入 **15 筆**名單（`MRN-DEMO-001`～`015`）及對應晚間批次報告，並含 **1 筆**無對應病歷之報告（`MRN-ORPHAN-NOMATCH`，供驗證 orphan 落地）；檢查日與追蹤旗標涵蓋「本月應追蹤」與非預設篩選可見之對照筆（實作見 `src/Web/src/views/CaseListView.vue`）。

## 六、風險與假設

- 外部系統（**EHR（LDCT 報告）**、**LLM（欄位擷取）**、**Power BI（報表與視覺化，連線本平台報表資料集）**、HIS、問卷、簡訊）介面與時程依院方與廠商協調；詳見 [design.md](openspec/changes/add-ldct-case-management-platform/design.md)、[proposal.md](openspec/changes/add-ldct-case-management-platform/proposal.md) 與 [openspec/project.md](openspec/project.md)。
