# data-sources — 資料來源

## ADDED Requirements

### Requirement: 健檢名單匯入（Health Screening Roster Import）

系統應（SHALL）支援自健檢中心／健管名單匯出檔匯入個案基本資料，並至少保存下列欄位：病歷號、姓名、性別、年齡、檢查日期、客戶名稱（公司團檢）、電話、手機。

#### Scenario: 匯入成功後可檢視欄位

- **WHEN** 使用者完成有效之健檢名單匯入
- **THEN** 該批個案於系統內可檢視上述欄位內容，且病歷號可作為個案主鍵與其他來源資料關聯

#### Scenario: 重複或格式錯誤之處理

- **GIVEN** 匯入檔含重複病歷號或必填欄位缺漏
- **WHEN** 執行匯入
- **THEN** 系統應（SHALL）阻擋或標示錯誤列，並保留可匯入列之結果或完整回滾（依院方選定策略於實作明確化），且使用者可取得錯誤說明

### Requirement: LDCT 影像報告匯入（LDCT Report Import）

系統應（SHALL）自 **EHR（電子病歷／院方臨床資訊系統）** 取得 LDCT 報告資料（介面型態依 `system-integrations` 與 `design.md`），並至少保存：是否有結節、結節數量、最大長度、是否需門診追蹤、3 個月追蹤、6 個月追蹤、1 年追蹤。若 EHR 提供之內容為非結構化敘述或文件，應經「LDCT 報告欄位 LLM 擷取」產生結構化欄位後，再與個案關聯。

對於已達院方定義之「**已發報告**」（正式發佈／已簽署等狀態）之 LDCT 報告，系統應（SHALL）依 [計畫書.md](../../../../../計畫書.md) 與 [design.md](../../design.md) 約定，於**該曆日之晚間排程批次**將資料**落地**至本平台，使關聯個案得以於**個案總清單**（`case-list`）依設計欄位與篩選使用；晚間之具體鐘點窗、時區、與 EHR 資料截止、失敗重試等，於實作配置與院方定案。

#### Scenario: 已發報告當日晚間納入個案清單

- **GIVEN** EHR 中某 LDCT 報告已達「已發報告」等院方定義狀態，且當日預定之**晚間批次**尚未執行或尚在處理中
- **WHEN** 該晚間批次／排程執行完成且報告與個案關聯成功
- **THEN** 報告欄位應已持久化，且個案總清單可檢視該筆（與健檢名單等來源之關聯規則一致）；若尚無對應個案，則依「無對應個案」策略處理

#### Scenario: 報告與個案關聯

- **GIVEN** 已存在對應病歷號之健檢／個案紀錄
- **WHEN** LDCT 報告資料自 EHR 進入本平台
- **THEN** 報告欄位應附加於該個案，並可供個案清單與追蹤邏輯使用

#### Scenario: 無對應個案

- **GIVEN** 自 EHR 取得之報告病歷號於系統中尚無個案
- **WHEN** 資料進入本平台
- **THEN** 系統應（SHALL）依可設定策略建立待併案／例外清單或拒絕寫入，並記錄供個管處理（策略於實作明確化）

### Requirement: LDCT 報告欄位 LLM 擷取（LLM Field Extraction from LDCT Report）

系統應（SHALL）透過 **LLM（大型語言模型）** 從 LDCT 報告之文字、敘述或文件內容中，截取並結構化與本系統一致之欄位，至少涵蓋與「LDCT 影像報告匯入」相同之語意：**是否有結節、結節數量、最大長度、是否需門診追蹤、3 個月追蹤、6 個月追蹤、1 年追蹤**（欄位定義與 `system-integrations` 之 LLM 服務整合一致）。

#### Scenario: 非結構化報告產生結構化欄位

- **GIVEN** 自 EHR 取得之 LDCT 報告為敘述文字、PDF 全文或其他需解析之格式
- **WHEN** 系統執行 LLM 擷取流程
- **THEN** 產生可與個案關聯之結構化欄位，並可供個案清單與追蹤邏輯使用

#### Scenario: 人工覆核與修正

- **WHEN** 具權限使用者修正 LLM 擷取之欄位
- **THEN** 以人工確認後之值作為業務有效值，並保留 LLM 原始輸出與修改紀錄供稽核（細粒度見 `design.md`）

#### Scenario: 低信心或無法擷取

- **GIVEN** LLM 無法可靠擷取某欄位或整份報告
- **WHEN** 擷取完成
- **THEN** 系統應（SHALL）標示待人工確認、缺漏或低信心狀態，且不得在未經人工確認下覆蓋已驗證資料

### Requirement: 個管師手動輸入（Case Manager Manual Entry）

系統應（SHALL）允許個案管理師手動輸入或編輯：報告判讀補充說明、個案追蹤狀態、電訪內容、聯繫紀錄（與「追蹤紀錄」能力搭配時，應避免重複定義；此處指可編修之敘述欄位與狀態）。

#### Scenario: 儲存與稽核

- **WHEN** 個管師儲存手動輸入內容
- **THEN** 內容應持久化並附帶操作者與時間（稽核），且具權限者可查閱

---

### Requirement: 問卷資料整合（Questionnaire Data）

系統應（SHALL）能整合護理師於平板填寫之「肺癌風險評估問卷」資料，來源為健檢問卷系統（匯入或 API，依 `system-integrations`）。

#### Scenario: 問卷與個案關聯

- **WHEN** 問卷資料進入系統
- **THEN** 應與正確個案（病歷號或院方約定之唯一鍵）關聯並可供清單與報表使用

---

### Requirement: 確診資料登錄（Diagnosis Data Entry）

系統應（SHALL）允許個管師自 HIS 查詢後，手動登錄確診相關資料，至少包含：是否確診肺癌、確診日期、確診方式、手術、切片、手術種類、病理報告、Stage。

#### Scenario: 確診欄位完整性

- **WHEN** 個管師標示為「確診肺癌」或相當旗標
- **THEN** 系統應（SHALL）提示或驗證關鍵欄位（如確診日期）之填寫，避免空值造成統計錯誤（細節可於實作與院方確認）
